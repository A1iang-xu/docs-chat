## 零、v3.2 → v3.3 升级动机

v3.2 的 13 项升级补齐了中文检索、语义分块、防幻觉等核心短板。但在逐模块审计中，发现以下 8 个环节存在 **"枪已上膛但扳机未扣"** 或 **"骨架正确但肌肉不协调"** 的问题：

| 环节 | v3.2 现状 | 核心缺陷 | 对最终效果的影响 |
|------|----------|----------|-----------------|
| 语义分块 | 固定 512 盲切后做后置合并 | 盲切已把句子拦腰截断，合并无法修复断点 | 语义分块的实际收益大打折扣 |
| 邻居扩展 | `_expand_neighbors()` 代码已写但未集成到 `chat_stream()` | 功能已实现但数据流中未调用 | 答案完整度优化未生效 |
| 忠实度验证 | 逐句串行调用 LLM（N 句 = N 次调用） | Token 消耗线性增长，延迟线性叠加 | 每次对话额外消耗 1600-4000 tokens |
| 忠实度反馈 | 检测到幻觉只发 warning，不拦截不重试 | 已标记的幻觉句子仍然推送给了用户 | 忠实度验证形同虚设 |
| 多轮对话 | 只截断最近 6 条，无摘要压缩 | 第 7 轮后丢失关键上下文（如"第三章"） | 多轮对话质量断崖式下降 |
| 查询分类 | 所有问题走相同管线 | 简单查询也触发 HyDE + CRAG，浪费 Token | 每次对话 5+N 次 LLM 调用 |
| 语义缓存预热 | 精确字符串匹配（O(1) 哈希） | 命中率极低，用户必须逐字重复 | 预热几乎无效 |
| 上下文 Token 预算 | 无限制拼接所有 chunk 全文 | 长文档可能超出有效注意力窗口 | 生成质量退化 |

v3.3 的使命：**不做新功能，只做管线精细化**——把每个环节从"骨架正确"提升到"肌肉协调、运行高效"。

## 二、P0 核心修复方案（概要）

### 3.3.1 忠实度批量验证

**现状诊断**：`faithfulness_checker.py` 对 LLM 生成的每句话发起一次独立的 LLM 非流式调用，且 `for` 循环串行执行。如果 LLM 生成了 8 句话，就是 8 次串行调用，每次约 200-500 tokens，总计 1600-4000 tokens 额外消耗，延迟线性叠加。

**最优解**：将 N 个句子打包成一次 LLM 调用，在一次请求中同时判断所有句子的忠实度。Prompt 中包含所有句子和所有 source docs，要求 LLM 输出一个 JSON 数组 `[{"index":0,"faithful":true,"evidence":"..."}, ...]`。同时在 `_split_into_statements()` 中引入两阶段过滤——先用 sentence-transformers 计算每句话与 source docs 的 Embedding 余弦相似度，高于阈值（如 0.7）的直接跳过，低于阈值的才进入 LLM 批量验证。

**关键实现点**：
- 修改 `_verify_sentence()` 为 `_verify_batch()`，一次调用处理所有句子
- 新增 `_prefilter_by_embedding()` 方法，用 Embedding 相似度做粗筛
- 两阶段过滤：Embedding 粗筛（快）→ LLM 精判（慢但准）

**量化预期**：延迟从 N×500ms 降至 1×500ms + Embedding 计算（~50ms），Token 消耗减少 70%。

---

### 3.3.2 忠实度反馈闭环

**现状诊断**：`rag_orchestrator.py` 第 132-137 行，忠实度验证发现幻觉后只记录 warning 日志和推送 SSE 事件，但 flagged 的句子仍然作为正常 token 推送给了用户。检测到了但没拦截，忠实度验证形同虚设。

**最优解**：在 `chat_stream()` 中，忠实度验证失败后不直接推送 flagged 句子。而是：
1. 将 flagged 句子及其被标记原因（如"该陈述无法在原文中找到支撑"）打包反馈给 LLM
2. 要求 LLM 重新生成这部分内容，明确禁止编造，要求引用原文
3. 对二次生成的结果再做一次忠实度验证
4. 如果二次仍失败，替换为降级话术："根据已有文档，该部分信息无法确认"

**关键实现点**：
- 修改 `chat_stream()` 的生成循环，在忠实度验证后增加 `_regenerate_if_unfaithful()` 分支
- 限制重试次数为 1 次，避免死循环
- 降级话术模板化，确保用户体验不中断

**量化预期**：幻觉句子真正被拦截，用户看到的答案修正率 +80%，幻觉率额外 -10~15%。

---

### 3.3.3 邻居扩展落地

**现状诊断**：`retrieval_service.py` 中已完整实现了 `_expand_neighbors()` 方法，且 `config.py` 中已定义 `CHUNK_NEIGHBOR_EXPAND=1`。但 `rag_orchestrator.py` 的 `chat_stream()` 中在获取 CRAG 结果后直接取 Top-K，没有调用邻居扩展。这是一处"枪已上膛但扳机未扣"的遗漏。

**最优解**：在 `rag_orchestrator.py` 的 `chat_stream()` 中，CRAG 过滤后、sandwich 排序前，插入邻居扩展调用。确保邻居 chunk 的分数降权（×0.8）且来源标记 `is_neighbor: true`，在 sandwich 排序时自然排在主 chunk 之后。

**关键实现点**：
- 在 `rag_orchestrator.py` 第 105-106 行之间插入邻居扩展调用
- 确保邻居 chunk 携带 `chunk_index` 和 `document_name` 字段以支持扩展
- 邻居分数降权，不影响主 chunk 的 sandwich 排序位置

**量化预期**：兑现 v3.2 的预期收益——答案完整度 +15~25%。

---

### 3.3.4 语义分块前置化

**现状诊断**：当前 `_semantic_rechunk()` 是在固定 512 字符盲切之后做的合并操作。如果盲切已经把一个句子从中间截断（"机器学习是人工智能的核心/技术之一"），合并操作只能把两个半句拼回去，但断点位置本身已经错了——"核心"和"技术之一"被分到了两个 chunk 再合并，中间的语义关联已经被打断。

**最优解**：将语义分块从"后置合并"改为"前置切分"。流程变为：
1. 先按句子拆分全文（`re.split(r'(?<=[。！？；\n])', text)`）
2. 用 sentence-transformers 计算相邻句子的 Embedding 余弦相似度
3. 在相似度骤降处（< threshold）标记断点
4. 在断点处切分，同时在累计长度超过 MAX_CHUNK_CHARS 时强制切分
5. 保留 overlap（前后各 1 句）

**与 v3.2 后置合并的关键区别**：前置切分在语义边界处切分，保证每个 chunk 是完整语义单元；后置合并在盲切后补救，只能减少碎片化但无法修复错误的断点。

**关键实现点**：
- 修改 `_semantic_rechunk()` 为 `_semantic_split()`，从"先切后合"改为"先分后切"
- 保留 `_split_long_section()` 的最大块限制逻辑作为安全网
- 在 PyPDF 路径和 MinerU 路径都使用前置切分

**量化预期**：语义分块实际收益从 v3.2 的 +10% 提升到 +20~30%。

## 三、P1 高优增强方案（概要）

### 3.3.5 查询分类路由

**现状诊断**：所有查询——无论是"文档的核心主题是什么"（需要深度检索）还是"文档有多少页"（元数据查询）——都走完全相同的昂贵管线。每次对话的 LLM 调用次数为：Query Rewrite(1) + HyDE(1) + CRAG 评估(1) + 可能重写(1) + 生成(1) + 忠实度验证(N) = 5+N 次。对于简单查询，这是巨大的浪费。

**最优解**：引入三级查询路由：
- **L1 元数据查询**（如"文档作者""文档页数"）：直接走 BM25 精确匹配，不触发向量检索、HyDE、CRAG
- **L2 概念查询**（如"解释一下注意力机制"）：走完整 RAG 管线但跳过 HyDE（概念查询与假设答案的语义鸿沟不大）
- **L3 综合推理**（如"比较第三章和第五章的性能指标"）：走完整管线，包括 HyDE + CRAG

分类器用一次轻量 LLM 调用（约 50 tokens）实现，延迟 < 500ms，远低于被跳过的环节。

**关键实现点**：
- 新增 `query_classifier.py`，用一次轻量 LLM 调用做三级分类
- 在 `rag_orchestrator.chat_stream()` 中根据分类结果动态跳过管线环节
- 分类结果缓存到对话上下文，同一轮对话中的追问复用分类

**量化预期**：简单查询 Token 消耗 -60%，延迟 -50%。

---

### 3.3.6 多轮对话摘要压缩

**现状诊断**：`_build_messages()` 中 `history[-6:]` 直接截断。在以下场景中会出问题：
- 用户在第 1 轮说"我关注的是第三章的性能指标"
- 第 2-6 轮围绕性能指标展开追问
- 第 7 轮问"那这个指标怎么优化"
- LLM 收到的历史中已经不包含"第三章"和"性能指标"，只能靠第 7 轮的问题猜测

**最优解**：引入 `ConversationSummaryMemory`——当历史消息超过阈值（如 4 条）时，触发一次轻量 LLM 调用，将历史压缩为一段 200 字以内的摘要，包含关键实体（如"第三章""性能指标"）、用户意图、已讨论的结论。摘要替换原始历史消息，释放上下文空间。

**关键实现点**：
- 新增 `conversation_compressor.py`，封装摘要压缩逻辑
- 在 `rag_orchestrator._build_messages()` 中，当 `len(history) > 4` 时触发压缩
- 压缩后的摘要作为 system prompt 的一部分注入，而非 user message
- 摘要缓存到对话上下文，避免同一轮对话中重复压缩

**量化预期**：长对话（10+ 轮）上下文保持率 +50%，关键实体丢失率 -80%。

---

### 3.3.7 上下文 Token 预算管理

**现状诊断**：`_build_messages()` 中 `"\n\n---\n\n".join(...)` 无限制拼接所有 chunk 的全文内容。5 个 chunk × 1024 字符 = 5120 字符 + 历史消息 + prompt 模板 + 生成预留。虽然 DeepSeek 上下文窗口是 128K，但 LLM 的有效注意力窗口远小于此，长上下文中间的 chunk 容易被忽略。

**最优解**：引入 Token 预算管理：
1. 计算总预算 = 模型上下文窗口（如 128K） - 生成预留（如 4096） - 系统提示词 - 历史消息 Token 数
2. 按 CRAG 分数按比例分配 Token 预算给每个 chunk：高分的 chunk 获得更多 Token 空间，低分的截断
3. 每个 chunk 的内容按 Token 预算截断，保留开头和结尾（对抗 Lost in the Middle 的另一个维度）
4. 如果总预算不足以容纳所有 chunk，按分数从低到高逐步丢弃

**关键实现点**：
- 新增 `token_budget.py`，封装 Token 计算与分配逻辑
- 使用 `tiktoken` 库做精确 Token 计数（而非字符数估算）
- 在 `_build_messages()` 中，chunk 拼接前先过 Token 预算分配

**量化预期**：关键信息密度 +30%，超出有效注意力窗口的风险清零。

---

### 3.3.8 语义缓存预热语义化

**现状诊断**：`semantic_cache.py` 的 `_warm_cache` 是一个内存 `dict`，key 是 query 原文。这意味着预热命中条件是用户逐字重复之前的问题——"总结第三章的核心技术指标"和"第三章主要讲了哪些技术指标"在预热缓存中是完全不同的 key，但语义几乎相同。预热命中率极低（< 5%），预热几乎无效。

**最优解**：将内存预热从精确字符串匹配改为向量近似匹配：
1. 启动时从 ChromaDB 加载最近 50 条热门查询及其向量
2. 用 FAISS 在内存中构建一个 L2 索引
3. 新查询先过 FAISS 做 O(log n) 近似匹配，找到 top-1 最相似查询
4. 如果相似度 ≥ 阈值（如 0.92），命中预热缓存，直接返回答案
5. 如果未命中，再走 ChromaDB 精确查询

**关键实现点**：
- 新增 `cache_warmup.py`，封装 FAISS 索引构建与查询
- 修改 `semantic_cache.py` 的 `lookup()` 方法，先查 FAISS 预热再查 ChromaDB
- 定期（如每小时）从 ChromaDB 同步最新热门查询到 FAISS 索引
- 内存预热大小可配置（默认 50 条），防止内存膨胀

**量化预期**：预热命中率从 <5% 提升到 30-50%，高频查询 TTFT 从 2-5s 降至 < 50ms 的比例大幅提升。
