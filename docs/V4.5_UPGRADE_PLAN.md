## F1: 混合检索权重自适应

### 目标
当前 BM25 + 向量检索的融合权重是固定的（`BM25_WEIGHT=0.3`），不同类型查询应使用不同权重。v4.5 根据查询分类结果动态调整权重，提升检索准确率 15%+。

### 现状分析
```python
# retrieval_service.py 当前实现
final_score = bm25_score * 0.3 + vector_score * 0.7  # 固定权重
```

### 实现方案
根据 v4.1 的 `query_classifier` 结果动态调整：
- **fact_lookup**（事实查询）：BM25 权重 ↑（关键词精确匹配更重要）→ `0.5 / 0.5`
- **concept_explain**（概念解释）：向量权重 ↑（语义相似更重要）→ `0.2 / 0.8`
- **synthesis**（综合查询）：均衡 → `0.3 / 0.7`
- **code**（代码查询）：BM25 权重 ↑（函数名/API 名精确匹配）→ `0.6 / 0.4`

### 文件变更
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `backend/app/services/retrieval_service.py` | 修改 | `search()` 接受 `query_type` 参数，动态计算权重 |
| `backend/app/services/rag_orchestrator.py` | 修改 | 将 `classify_result["query_type"]` 传入检索 |
| `backend/app/core/config.py` | 修改 | 新增权重映射配置 |
| `backend/tests/test_hybrid_weights.py` | 新建 | 权重自适应测试 |

## F2: 查询改写与 HyDE

### 目标
用户查询常过于简短或模糊（如"那个 ref"），直接检索效果差。通过 LLM 改写查询 + HyDE（Hypothetical Document Embeddings）提升召回率 20%+。

### 实现方案
**两阶段改写**：
1. **查询改写**：结合对话历史，将指代消解为完整查询
   - "那个 ref" + 历史"什么是 Vue 3" → "Vue 3 中的 ref 是什么"
2. **HyDE**：让 LLM 先生成一个假设性回答，用回答的向量去检索（而非原始查询）
   - 查询 "ref 和 reactive 区别" → LLM 生成假设回答 → 用回答向量检索

### 文件变更
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `backend/app/services/query_rewriter.py` | 修改 | 新增 `rewrite_with_history()` 和 `hyde_generate()` |
| `backend/app/services/rag_orchestrator.py` | 修改 | 在分类后、检索前插入改写 + HyDE |
| `backend/app/core/config.py` | 修改 | `HYDE_ENABLED`, `HYDE_MAX_TOKENS` |
| `backend/tests/test_query_rewriter.py` | 新建 | 改写/HyDE 测试 |

## F3: 回答后处理管线

### 目标
LLM 生成的原始回答常有格式问题（多余空行、引用编号不连续、代码块未闭合），后处理管线提升输出质量。

### 实现方案
新建 `backend/app/services/answer_postprocessor.py`，管线化处理：
1. **引用对齐**：确保 `[1]` `[2]` 编号与 sources 列表一致
2. **代码块修复**：未闭合的 ``` 自动补全
3. **冗余清理**：移除 "根据文档内容" 等无意义前缀
4. **格式规范化**：统一标题层级、列表缩进

### 文件变更
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `backend/app/services/answer_postprocessor.py` | 新建 | 后处理管线 |
| `backend/app/services/rag_orchestrator.py` | 修改 | 生成后调用后处理 |
| `backend/tests/test_postprocessor.py` | 新建 | 后处理测试 |

## F5: CI/CD 流水线

### 目标
GitHub Actions 自动化测试 + lint，每次 push 自动验证代码质量。

### 实现方案

  frontend-build:
