# DocsChat v4.3 升级方案 — 质量验证与产品化打磨

> **面向执行**：本文档供 Claude Code 逐任务实施。
> **前置版本**：v4.2（bug 修复 + 测试覆盖 + 文档更新）
> **日期**：2026-06-26

---

## 一、定位与目标

v4.0 把管线接进了开发者文档场景，v4.1 深化了三项算法，v4.2 修复了 bug 并补了测试。但系统仍缺三个关键能力：**无法量化证明回答质量、无法直观展示管线状态、无法让新人 5 分钟跑通**。v4.3 聚焦这三点，把系统从"能跑"推进到"可验证、可展示、可交付"。

**三项任务**：
- D1 RAGAS 自动化评估：用标准指标量化回答质量，生成评估报告
- D2 前端可观测面板：把后端已有的 metrics 数据可视化展示
- D3 端到端集成测试 + 预置示例库：入库真实文档、跑通完整问答链路、新人一键体验

---

## 二、当前代码现状诊断

### 2.1 无质量评估体系

系统有忠实度检查（`faithfulness_checker`）和用户反馈（`feedback_service`），但缺少**标准化的离线评估框架**。面试官问"你的 RAG 系统回答质量如何"时，只能靠主观感受，无法给出 faithfulness/context_precision/answer_relevancy 等标准指标数值。

### 2.2 可观测数据有但不可见

后端 `metrics_service.py` 已实现 P50/P95 延迟、缓存命中率、忠实度警告率等指标聚合，`GET /api/stats` 已暴露数据。但前端没有对应的可视化面板——指标数据存在却无法被直观看到。

### 2.3 无预置示例库

`docker-compose.yml` 已加 Crawl4AI sidecar，但启动后是空库，用户必须自己上传 PDF 或输入 URL 才能体验。没有"开箱即用"的示例文档库，新人无法 5 分钟跑通。

## 三、任务分解

### 任务 D1：RAGAS 自动化评估

**目标**：引入 RAGAS 评估框架，对系统的 faithfulness、context_precision、context_recall、answer_relevancy 四项标准指标进行离线评估，生成可量化的质量报告。

**新增文件**：`backend/app/services/evaluation_service.py`、`backend/app/api/evaluation.py`、`backend/tests/eval_dataset.json`

**当前实现**：无评估框架。`faithfulness_checker` 是生成时的在线检查，不是离线评估。

**目标实现**：

#### D1.1 评估数据集

`backend/tests/eval_dataset.json`：预置 10-15 条开发者文档典型问答对（query + expected_answer + expected_sources），覆盖事实/概念/综合三类查询：

```json
[
  {
    "query": "what is ref() in Vue",
    "expected_answer": "ref() is a function that creates a reactive reference...",
    "expected_keywords": ["reactive", "reference", ".value"],
    "query_type": "concept_explain"
  },
  {
    "query": "how to use computed in Vue 3",
    "expected_answer": "Use the computed() function...",
    "expected_keywords": ["computed()", "readonly", "cache"],
    "query_type": "synthesis"
  }
]
```

#### D1.2 评估服务

`backend/app/services/evaluation_service.py`：

```python
class EvaluationService:
    """RAGAS 风格的离线评估服务。

    评估指标:
    - faithfulness: 答案是否忠于检索到的上下文（不幻觉）
    - context_precision: 检索的上下文是否精准（相关上下文排名靠前）
    - context_recall: 检索是否覆盖了回答所需的全部信息
    - answer_relevancy: 答案是否切题
    """

    async def evaluate_single(self, query: str, answer: str,
                               contexts: list[str],
                               expected: dict) -> dict:
        """评估单条问答，返回各指标分数 (0-1)。"""
        return {
            "faithfulness": await self._eval_faithfulness(answer, contexts),
            "context_precision": await self._eval_context_precision(query, contexts),
            "context_recall": await self._eval_context_recall(expected, contexts),
            "answer_relevancy": await self._eval_answer_relevancy(query, answer),
        }

            "total": len(results),
            "avg_faithfulness": ...,
            "avg_context_precision": ...,
```

评估方法有两种实现路径：
- **路径 A（推荐）**：用 LLM 评估（复用 DeepSeek），模拟 RAGAS 的 LLM-as-judge 方法，不依赖 ragas 库
- **路径 B**：安装 `ragas` 库，但 ragas 依赖较重且与 LangChain 版本可能冲突

v4.3 第一步用路径 A——自己实现 LLM-as-judge 逻辑，避免依赖冲突，也更可控。

**改造文件**：`frontend/src/views/StatsView.vue`（新建）、`frontend/src/router` 或 `App.vue`（加导航入口）、`frontend/src/utils/api.ts`（加 stats API 调用）

**目标实现**：

#### D2.1 StatsView 页面

`frontend/src/views/StatsView.vue`：

## 七、实施顺序

```
D3 预置示例库 + 端到端测试（先做，为 D1 评估提供数据基础）
  ↓
D1 RAGAS 自动化评估（依赖有入库数据才能跑评估）
  ↓
D2 前端可观测面板（独立，可并行）
```

D3 先做的原因：D1 评估需要系统里有入库的文档才能跑；D3 的预置库脚本正好提供这个基础。
