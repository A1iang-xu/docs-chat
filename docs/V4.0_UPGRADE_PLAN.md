## 一、定位与目标

v3.3 把 RAG 管线打磨到"骨架完整、肌肉协调"——查询分类、FAISS 预热、忠实度反馈闭环、Token 预算都在 `rag_orchestrator.py` 里跑通了。但整套系统仍是一个"上传 PDF → 问答"的通用 demo：摄取层死耦合本地文件路径，向量库没有"库"的概念，引用只能定位到"第 X 页"。

v4.0 把这套管线接进一个真实场景——**面向开发者的官方文档问答**。开发者粘个 `vuejs.org` 或 `fastapi.tiangolo.com` 链接就能建库问答，比传 PDF 有说服力得多。

**三支柱均衡推进**：
- 支柱 A（场景定制）：URL 自动抓取入库、代码感知分块、多库命名空间、URL 引用溯源
- 支柱 B（生产工程化）：可观测闭环、限流修复、增量去重、一键启动+在线 Demo
- 支柱 C（高级 RAG）：用户反馈闭环、语义化查询分类、代码子索引、Agentic 多跳

## 三、目标架构

在 v3.3 成熟查询管线之上，新增"双源摄取"与"按库命名空间"两条主线，可观测与反馈两条暗线接回检索与生成环节。

```
摄取层（双源）                         查询层
┌─────────────────────┐              ┌──────────────────────────────────┐
│ 官方文档 URL          │             │ 用户 + 库选择器                    │
│  ↓                   │             │  ↓                                │
│ Crawl4AI v0.5.0      │             │ 查询分类路由                        │
│ Sitemap优先+同域BFS   │             │  ↓                                │
│ Playwright·Fit MD    │             │ 语义缓存(FAISS预热) ─命中→ 答案     │
│  ↓                   │             │  ↓未命中                           │
│ 代码感知分块          │             │ 混合检索(where=library)            │
│ MarkdownHeader+代码保护│            │  ↑              ↑                 │
│  ↓                   │             │ ChromaDB       BM25(按库增量)       │
│ PDF上传(保留)→MinerU  │            │  ↓                                │
│  ↓                   │             │ RRF→Reranker→CRAG→LLM生成         │
│  ┌────────────────┐  │             │  ↓                                │
│  │ ChromaDB       │←─┼────────────│ 忠实度反馈闭环                      │
│  │ library命名空间 │  │             │  ↓                                │
│  └────────────────┘  │             │ 答案+URL引用溯源                   │
│  ↓                   │             │  ↓                                │
│ BM25(按库增量)        │             │ 用户反馈 👍/👎 ─数据飞轮→ 检索     │
└─────────────────────┘              └──────────────────────────────────┘
                                        ↑ 可观测指标采集
```

**保留不动**：v3.3 的查询分类、HyDE、RRF、Reranker、CRAG、忠实度反馈闭环、Token 预算、FAISS 预热。v4.0 只在检索入口加 `where` 过滤，不动核心管线。

## 四、任务分解

### P0 — 场景落地（让它"是"文档助手）

#### 任务 A1：URL 自动抓取入库

**目标**：输入官方文档站地址，自动抓取页面、解析为 Markdown、分块入库，产出可检索的知识库。

**抓取策略**：Sitemap 优先 + 同域受控 BFS 爬取。优先解析站点 `sitemap.xml` 作为 URL 种子清单；无 sitemap 则退化为同域 BFS，限制最大页数与深度，遵守 `robots.txt`。

**工具选型**：Crawl4AI v0.5.0（pip 包 `crawl4ai`）。核心能力：
- BFS 深度爬取带 `max_pages` 上限（受控）
- Playwright JS 渲染（适配 Vitepress/Docusaurus/Nextra 等 SPA 文档站）
- `PruningContentFilter` 产出 Fit Markdown（去除导航/广告/页脚噪声）

**新增文件**：`backend/app/services/web_ingestion_service.py`

实现一个 `WebIngestionService` 类：
- `async def ingest_url(self, url: str, library_slug: str, version: str = "latest") -> str`：主入口，返回 job_id
- `async def _resolve_sitemap(self, base_url: str) -> list[str]`：解析 sitemap.xml，无则返回空列表
- `async def _crawl(self, urls: list[str], max_pages: int) -> list[dict]`：调用 Crawl4AI 抓取，返回 `[{url, markdown, title}]`
- `async def _run_job(self, job_id: str, url: str, library_slug: str, version: str)`：异步执行抓取→分块→入库

复用 v3.3 已有的异步任务骨架：走 `ingestion_service` 的 `JobStatus` 状态机与 `INGESTION_MAX_CONCURRENT_JOBS` 信号量，质量门禁 `quality_gate.validate` 在抓取产物上同样生效。

#### 任务 A2：代码感知分块

**目标**：分块时不截断代码块，保留代码完整性，扩展 chunk 元数据支持文档站场景。

**改造文件**：`backend/app/services/document_service.py` + 新增 `backend/app/services/markdown_chunker.py`

**当前实现**（`document_service.py` 第 14-30 行）：`DocumentChunk` 字段为 `chunk_id/content/document_name/page/chunk_index/metadata`。

**目标实现**：

1. 扩展 `DocumentChunk`，新增可选字段：
```python
class DocumentChunk:
    def __init__(self, chunk_id, content, document_name, page, chunk_index,
                 metadata=None, source_url="", library="", version="",
                 heading_path="", code_language="", is_code_block=False):
        # ... 原有字段 ...
        self.source_url = source_url
        self.library = library

        self.heading_path = heading_path
        self.code_language = code_language
```

2. 新增 `backend/app/services/markdown_chunker.py`，实现代码感知分块：
- 第一阶段：用 `langchain_text_splitters.MarkdownHeaderTextSplitter` 按 h1/h2/h3 切出语义单元，保留标题路径

**验收标准**：
- 包含 ` ```python\nprint("hello")\n``` ` 的 Markdown 分块后，代码块完整存在于单个 chunk 中
- `DocumentChunk.is_code_block=True` 且 `code_language="python"`

#### 任务 B3：增量摄取与去重

**目标**：同一页面重新抓取不产生重复 chunk，增量更新只处理变更页。

**改造文件**：`document_service.py`、`mineru_document_service.py`、`vector_store.py`

**当前实现**：`document_service.py` 第 82 行 `chunk_id=uuid4().hex`，`mineru_document_service.py` 第 360 行 `_make_chunk` 同样 `uuid4().hex`——随机 ID，重复入库。

**目标实现**：

1. chunk_id 改为确定性：
```python
import hashlib
def _deterministic_chunk_id(library, source_url, heading_path, content):
    raw = f"{library}|{source_url}|{heading_path}|{content[:200]}"
    return hashlib.sha256(raw.encode()).hexdigest()[:32]
```

2. `vector_store.add_chunks` 改为 `upsert`（ChromaDB `collection.upsert(ids=..., documents=..., metadatas=...)`）。

3. `web_ingestion_service` 摄取前按 `library + source_url` 查已有块，做差量，跳过未变更页。

**验收标准**：
- 同一 URL 两次入库，chunk 数量不变（upsert 覆盖）
- 增量更新只处理变更页，未变更页跳过
