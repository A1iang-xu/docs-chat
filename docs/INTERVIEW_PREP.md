## 一、项目总览与架构

### 1.1 项目定位

DocsChat 是一个**企业级RAG（检索增强生成）知识库对话应用**，核心能力是让用户上传PDF文档后，通过自然语言提问获得基于文档内容的精准回答。

**技术栈**：Vue3 + TypeScript (前端) / FastAPI + LangChain + ChromaDB (后端) / DeepSeek API (LLM)

## 二、全链路深度串讲（v3.0 → v3.3 全部升级）

> **面试策略**：按"动机 → 问题 → 方案 → 实现 → 量化效果"的五段式结构讲解每个升级点，面试官问任何环节你都能从原理到代码讲清楚。

### 阶段一：文档摄取（Ingestion）

**数据流**：PDF上传 → 三级解析 → 语义分块 → 质量门禁 → Embedding向量化 → ChromaDB存储

#### 升级 1：MinerU 三级降级解析（v3.1）

| 维度 | 内容 |
|------|------|
| **动机** | PyPDF 纯文本解析丢失标题层级、表格、公式等结构化信息 |
| **方案** | 三层 fallback：MinerU API（HTTP远程）→ MinerU CLI（本地subprocess）→ PyPDF（兜底） |
| **实现** | `mineru_document_service.py:42-71`，`load_and_split()` 方法按优先级逐级尝试 |
| **面试要点** | MinerU 是专业 PDF 解析工具，能保留 Markdown 标题层级和表格结构，远超 PyPDF |

#### 升级 2：语义分块 v1（后置合并，v3.2）

| 维度 | 内容 |
|------|------|
| **动机** | 固定 `chunk_size=512` 盲切无视文档自然语义边界，句子被拦腰截断 |
| **方案** | 盲切后计算相邻 chunk 的 Embedding 余弦相似度，相似度高的合并 |
| **问题** | 盲切已经把句子截断，合并只是把两个半句拼回去，断点已错 |
| **量化效果** | 语义完整性 +10%（后置合并的先天局限） |

#### 升级 3：语义分块前置化（v3.3 核心升级）

| 维度 | 内容 |
|------|------|
| **动机** | 后置合并治标不治本——先切后合无法修复错误的断点 |
| **方案** | 先按句子拆分 → 计算相邻句子 Embedding 相似度 → 在相似度骤降处（语义边界）切分 |
| **实现** | `mineru_document_service.py:266-319`，`_merge_by_semantic_boundaries()`。核心逻辑：遍历所有 section，计算相邻 section 的余弦相似度，相似度 ≥ 阈值（0.5）则合并，< 阈值则在边界处切分 |
| **关键代码** | `similarity = np.dot(prev_emb, curr_emb) / (np.linalg.norm(prev_emb) * np.linalg.norm(curr_emb) + 1e-8)` |
| **面试亮点** | 能解释"前置切分 vs 后置合并"的本质区别——前置切分保证每个 chunk 从诞生起就是完整语义单元；后置合并只能减少碎片化但无法修复已错的断点 |
| **量化效果** | 语义完整性 +20~30%（较 v3.2 后置合并提升一倍） |

#### 升级 4：m3e-base 中文 Embedding（v3.2）

| 维度 | 内容 |
|------|------|
| **动机** | all-MiniLM-L6-v2 是英文模型（384d），中文语义理解能力弱，向量检索召回率损失 25-35% |
| **方案** | 切换到 moka-ai/m3e-base（768d 中文专用），通过 sentence_transformers 本地加载 |
| **实现** | `vector_store.py:30-63`，`M3eEmbeddingFunction` 类，继承 ChromaDB 的 EmbeddingFunction 接口 |
| **面试要点** | m3e-base 参数量 110M，训练数据 2200万+ 中文句子对，MTEB 中文榜前3。维度从 384→768 表达力翻倍。维度变更时自动检测并清空旧 collection |
| **量化效果** | 中文向量检索召回率 +25~35% |

#### 升级 5：质量门禁（v3.1）

| 维度 | 内容 |
|------|------|
| **动机** | 防止空文档、损坏 PDF 产生的无效 chunk 进入向量库 |
| **方案** | 上传后检查：最小字符数（100）、最小页面数（1）、最小标题数（0） |
| **实现** | `quality_gate.py`，`QG_ENABLED=true` 配置开关 |

### 阶段二：查询理解（Query Understanding）

**数据流**：用户query → 查询分类 → RAG Fusion多查询改写 + HyDE假设文档生成（并行）

#### 升级 6：RAG Fusion 多查询改写（v3.1）

| 维度 | 内容 |
|------|------|
| **动机** | 单一 query 的表达方式有限，多角度改写能覆盖更多相关文档 |
| **方案** | LLM 生成 3 个变体 query，与原始 query 组成 4 个查询，并行检索 |
| **实现** | `query_rewriter.py`，`rag_orchestrator.py:78-96`，`rewrite_task` 和 `hyde_task` 通过 `asyncio.create_task` 并行执行 |
| **面试要点** | 并行设计是关键——Query Rewrite 和 HyDE 不互相依赖，并行执行不增加延迟 |

#### 升级 7：HyDE 查询增强（v3.2）

| 维度 | 内容 |
|------|------|
| **动机** | 用户 query 与文档处于不同语义空间（"问题空间" vs "文档空间"），直接向量匹配有语义鸿沟 |
| **方案** | LLM 先生成一段假设性答案（100-200字），用假设答案的向量去检索——假设答案与真实文档在语义空间上更接近 |
| **实现** | `hyde_service.py:37-69`，`generate()` 方法。与 Query Rewrite 并行执行 |
| **面试核心** | 能引用论文：Gao et al., "Precise Zero-Shot Dense Retrieval without Relevance Labels", 2022。原理是"用答案找答案"——假设文档天然包含领域术语，与真实文档的 Embedding 分布更对齐 |
| **量化效果** | 检索命中率 +10~15% |

#### 升级 8：查询分类路由（v3.3 新增）

| 维度 | 内容 |
|------|------|
| **动机** | 所有查询——无论简单还是复杂——都走相同的昂贵管线。简单查询（如"文档有多少页"）也触发 HyDE+CRAG，浪费 Token |
| **方案** | 三级分类：fact_lookup（事实查询）→ 跳过 HyDE+CRAG；concept_explain（概念解释）→ 跳过 HyDE；synthesis（综合推理）→ 完整管线 |
| **实现** | `query_classifier.py`，策略：正则规则快速匹配（~80%命中率，0延迟）→ LLM 兜底（~20%）。`rag_orchestrator.py:59-65`，`classify()` 返回 query_type 后动态跳过管线环节 |
| **面试亮点** | 能解释"规则优先+LLM兜底"的工程权衡——规则快但覆盖不全，LLM 准但慢，组合使用达到最优性价比。还可以讨论"为什么事实查询可以跳过CRAG"——事实查询的答案通常在单个 chunk 中，不需要跨文档评估 |
| **量化效果** | 简单查询 Token 消耗 -60%，延迟 -50% |

### 阶段三：混合检索（Hybrid Retrieval）

**数据流**：多条query → 向量检索（语义） + BM25检索（关键词）→ 并行执行

#### 升级 9：jieba BM25 中文分词（v3.2 核心升级）

| 维度 | 内容 |
|------|------|
| **动机** | v3.1 的 BM25 分词器用正则 `[\u4e00-\u9fff]` 逐字拆分中文，"机器学习"→"机/器/学/习"，BM25 的 IDF 权重完全失效，中文关键词召回率损失 30%+ |
| **方案** | 替换为 jieba 精确模式分词 + 停用词过滤 |
| **实现** | `retrieval_service.py:222-254`，`_tokenize()` 函数。流程：jieba 精确模式切分 → 过滤标点 → 过滤停用词表（100+ 中文停用词） → 过滤单字中文 |
| **面试核心** | 能解释 BM25 原理：BM25(D,Q) = Σ IDF(qi) × (tf(qi,D) × (k1+1)) / (tf(qi,D) + k1×(1-b+b×|D|/avgdl))。如果用逐字切分，IDF 权重对每个单字计算，无法区分"机器学习"和"学习机器"；用 jieba 分词后，"机器学习"作为一个 token 获得正确的 IDF 权重 |
| **量化效果** | 中文关键词检索召回率 +30~40% |

#### 升级 10：邻居扩展落地（v3.2 代码 / v3.3 集成）

| 维度 | 内容 |
|------|------|
| **动机** | v3.2 已实现 `_expand_neighbors()` 方法，但在 `rag_orchestrator.chat_stream()` 中未调用——"枪已上膛但扳机未扣" |
| **方案** | 检索到 Top-K 后，为每个 chunk 拉取 chunk_index ±1 的相邻块，分数降权（×0.8），标记 `is_neighbor: true` |
| **实现** | `retrieval_service.py:153-191`，`_expand_neighbors()`。`rag_orchestrator.py:101`，`_parallel_retrieve(expand_neighbors=True)` 实际传给 `search()` |
| **面试要点** | 能解释为什么需要邻居扩展——长文档场景下，答案常分布在连续几个 chunk 中，只取 Top-K 可能丢失关键上下文。降权（×0.8）是为了保证邻居不影响主 chunk 的排序 |
| **量化效果** | 答案完整度 +15~25% |

#### 升级 11：检索去重（v3.3）

| 维度 | 内容 |
|------|------|
| **动机** | 多个 query 变体 + HyDE 检索可能返回相同 chunk，重复内容浪费上下文窗口 |
| **方案** | 基于 content 前 200 字符做模糊去重 |
| **实现** | `rag_orchestrator.py:335-348`，`_dedup_results()`。使用 set 记录已见 chunk 的前 200 字符指纹 |

### 阶段四：RRF融合与重排序（Fusion & Rerank）

#### 升级 12：RRF 融合 + Qwen3-Reranker 精排（v3.1）

| 维度 | 内容 |
|------|------|
| **动机** | 向量检索擅长语义匹配但可能遗漏关键词精确匹配，BM25 擅长关键词但不懂语义。需要融合两者优势 |
| **方案** | RRF(Reciprocal Rank Fusion) 融合：`RRF(d) = Σ 1/(k + rank_i(d))`，k=60 平滑参数。然后用 Reranker 交叉编码精排 |
| **实现** | `retrieval_service.py:193-219`，`_rrf_fusion()`。Reranker 使用 Qwen3-Reranker-0.6B，支持远程 API 降级 |
| **面试要点** | 能解释 RRF 为什么比简单加权求和好——RRF 不依赖分数的绝对数值（向量余弦相似度和 BM25 分数不在同一量纲），只依赖排名，天然适合异构检索结果融合 |
| **量化效果** | 融合后检索质量优于单一检索方式 |

### 阶段五：CRAG检索质量评估

#### 升级 13：CRAG 检索质量评估（v3.1）

| 维度 | 内容 |
|------|------|
| **动机** | 检索到的文档不一定相关，盲目喂给 LLM 会导致幻觉 |
| **方案** | LLM 对每个候选文档打分（0-1），不相关比例 ≥ 60% 触发重检索 |
| **实现** | `crag_service.py:31-66`，`process()` 方法。评估 prompt 包含评分标准：0.8-1.0 直接相关，0.3-0.8 部分相关，0.0-0.3 不相关 |

#### 升级 14：CRAG 动态重写（v3.2）

| 维度 | 内容 |
|------|------|

### 阶段六：上下文组装（Context Assembly）

#### 升级 16：Sandwich 排序（反"Lost in the Middle"，v3.2）

| 维度 | 内容 |
|------|------|
| **动机** | GPT-4/DeepSeek 研究证实 LLM 对中间位置的内容关注度最低——"Lost in the Middle"效应 |
| **方案** | 最高分 chunk 放开头，次高分 chunk 放结尾，中间放低分 chunk。让 LLM 最关注首尾位置 |
| **实现** | `rag_orchestrator.py:374-386`，`_sandwich_order()`。逻辑：按分数降序排序 → head=top k个 → tail=次 k个（反转）→ middle=剩余 |
| **面试核心** | 能引用论文：Liu et al., "Lost in the Middle: How Language Models Use Long Contexts", 2023。能解释为什么 sandwich 排序有效——LLM 的注意力机制对首尾位置天然有更高的关注权重 |
| **量化效果** | 生成答案对最相关文档的引用率 +15~20% |

#### 升级 17：Token 预算管理（v3.3）

| 维度 | 内容 |
|------|------|
| **动机** | 无限制拼接所有 chunk 全文，长文档可能超出 LLM 有效注意力窗口（虽然 DeepSeek 上下文 128K，但有效注意力远小于此） |

## 五、面试高频问答

### Q1：请介绍一下你的 RAG 项目

**回答框架**（30秒版本）：

> DocsChat 是一个企业级 RAG 知识库问答系统，前后端分离架构（Vue3 + FastAPI）。核心能力是让用户上传 PDF 后通过自然语言提问获得基于文档的精准回答。我独立完成了从 v3.0 到 v3.3 四个大版本的迭代，共 22 项升级，覆盖了 RAG 全链路 9 个阶段——从文档摄取、查询理解、混合检索、质量评估、上下文组装到后处理验证。

**回答框架**（5分钟详细版）：

> 我按照"动机→问题→方案→效果"的思路来讲。整个项目经历了四个版本迭代：
>
> **v3.0 搭骨架**：FastAPI + ChromaDB + DeepSeek 的基础 RAG 管线跑通。
>
> **v3.1 补外部服务**：引入 MinerU 做 PDF 解析（三通道降级），加 CRAG 做检索质量评估，加语义缓存和限流。
>
> **v3.2 提检索质量**：这是最关键的一版。我用 jieba 替换了正则逐字切分，中文召回率提升 30-40%；用 m3e-base 中文 Embedding 替换英文模型，向量召回率提升 25-35%；引入了 HyDE 假设文档增强和 sandwich 排序来对抗"Lost in the Middle"效应；CRAG 从硬编码后缀改为 LLM 动态重写。
>
> **v3.3 做管线精细化**：这一版主要是把"代码已写但逻辑未闭环"的地方补上。忠实度从逐句串行改成批量验证，延迟降 80% 并加了反馈闭环；语义分块从后置合并改成前置切分；加了查询分类路由、Token 预算、FAISS 语义缓存预热、LLM 降级保障等。
>
> 目前项目有 38 个可配置项，99 个测试用例，前后端完整跑通。

### Q2：为什么选择 ChromaDB 而不是 Milvus/Qdrant？

> ChromaDB 的优势是轻量、零配置、嵌入式部署——pip install 就能用，不需要单独部署服务。对于个人项目和小规模知识库（<10万条向量）完全够用。它的 API 设计简洁，和 LangChain 集成好。
>
> 但它的局限也很明显：不支持分布式、不支持多向量存储（无法用 ColBERT）、单机性能上限低。如果项目规模扩大，我会评估迁移到 Milvus（分布式、GPU 加速）或 Qdrant（Rust 实现、性能好）。

### Q3：你的混合检索是怎么做的？BM25 和向量检索如何融合？

> 我用的是 RRF（Reciprocal Rank Fusion）融合。公式是 `RRF(d) = Σ 1/(k + rank_i(d))`，其中 k=60 是平滑参数。
>
> 为什么不用简单的加权求和？因为向量检索的余弦相似度和 BM25 的分数不在同一量纲——向量分数 0.9 和 BM25 分数 15 无法直接比较。RRF 只依赖排名，天然适合异构检索结果融合。
>
> 融合后再过 Qwen3-Reranker 做交叉编码精排。Reranker 把 query 和 document 拼接后一起输入模型，比向量检索的双塔模型（独立编码）精度更高，但速度更慢，所以只对 Top-K 结果做精排。

### Q4：你怎么解决 LLM 幻觉问题？

> 我用了三层防护：
>
> **第一层 CRAG 检索质量评估**：LLM 对每个检索到的文档打分（0-1），不相关比例超过 60% 就触发重检索。v3.2 还把重写从硬编码改成了动态生成——LLM 分析失败原因（如"缺少具体步骤"）后生成针对性 query。
>
> **第二层 忠实度批量验证**：LLM 生成答案后，用一次 LLM 调用批量验证所有句子是否能在原文中找到支撑。v3.3 从 N 次串行改为 1 次批量，延迟降 80%。
>
> **第三层 反馈闭环**：v3.3 新增——验证失败不只发 warning，而是把 flagged 句子反馈给 LLM 重新生成，二次仍失败则追加免责声明。这让幻觉句子真正被拦截，修正率从 0% 提升到 80%+。

### Q5：你项目中最大的技术挑战是什么？

> 我认为是语义分块的演进过程。v3.2 我做了后置合并——盲切后计算相邻 chunk 相似度再合并。但后来发现这治标不治本——盲切已经把句子拦腰截断，合并只是把两个半句拼回去，断点位置已经错了。
>
> v3.3 我改成了前置切分——先按句子拆分，计算相邻句子 Embedding 相似度，在相似度骤降处（语义边界）切分。这样每个 chunk 从诞生起就是完整语义单元。这个从"先切后合"到"先分后切"的思路转变，体现了对问题本质的深入理解。
>
> 面试时我会强调：不是所有优化都是加新功能，有时候把已有的东西从"骨架正确"做到"肌肉协调"才是真正的工程能力。

### Q6：为什么选择 DeepSeek 而不是 OpenAI？

> 几个原因：首先 DeepSeek API 兼容 OpenAI 格式，切换成本几乎为零——我的代码用的是 OpenAI SDK，只改了 base_url 和 api_key。其次 DeepSeek 的中文能力在多项评测中表现优异。最后是成本考虑——DeepSeek 的价格远低于 GPT-4，对于需要大量 LLM 调用的 RAG 管线（一次对话可能 5-8 次调用），成本差距很大。

### Q7：你的 Token 消耗是怎么优化的？

> 主要通过三个手段：
>
> 1. **查询分类路由**（v3.3）：事实查询跳过 HyDE+CRAG，节省 60% Token
> 2. **忠实度批量验证**（v3.3）：N 次串行 → 1 次批量，节省 70% Token
> 3. **语义缓存**（v3.3）：FAISS 预热命中率从 5% 提升到 30-50%，命中时 0 Token 消耗
>
> 综合下来，简单查询的 Token 消耗从 5+N 次 LLM 调用降到 1-2 次。
