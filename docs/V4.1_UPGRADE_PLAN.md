# DocsChat v4.1 升级方案 — P2 算法深化

> **面向执行**：本文档供 Claude Code 逐任务实施。每个任务标注涉及文件、当前实现、目标实现与验收标准。
> **前置版本**：v4.0（场景落地 + 工程化，已完成）
> **日期**：2026-06-25

---

## 一、定位与目标

v4.0 把管线接进了开发者文档场景——URL 自动抓取入库、代码感知分块、多库命名空间、可观测闭环、反馈闭环全部闭合。v4.1 聚焦 v4.0 计划中标注为 P2 的三项算法能力，目标是让回答质量真正上一个台阶。

**核心矛盾**：v4.0 的场景是开发者技术文档（Vue/FastAPI/LangChain），大量查询是英文（"how to use ref()"、"what is dependency injection"），但查询分类器仍是纯中文关键词正则，对英文查询几乎不生效。这是 v4.1 要解决的第一件事。

**三项任务**：
- C1 语义化查询分类：从中文关键词正则升级为双语 few-shot + embedding 分类
- C2 代码子索引：代码块单独索引，代码意图查询路由到代码子索引
- C4 Agentic 多跳检索：跨版本/跨库对比查询的 query planner + 多步综合

---

## 二、当前代码现状诊断

### 2.1 查询分类器：中文关键词，英文查询失效

`backend/app/services/query_classifier.py`：

第 26-36 行，三个正则模式全是中文：
```python
_FACT_PATTERNS = re.compile(
    r"(是多少|哪天|几月|几点|谁|什么人|什么时间|什么时候|在哪|哪里"
    r"|代码|参数|版本|配置|几行|多少页|第几章|具体数字|截至)"
)
_CONCEPT_PATTERNS = re.compile(
    r"(是什么|什么意思|如何理解|定义|概念|原理|为什么|原因|解释|介绍)"
)
_SYNTHESIS_PATTERNS = re.compile(
    r"(如何|怎么|怎样|比较|区别|vs|对比|分析|总结|归纳|梳理|评估"
    r"|优缺点|利弊|方案|建议|推荐|步骤|流程|方法)"
)
```

用户查询 "how to use ref()" 时：
- fact_count=0, concept_count=0, synthesis_count=0
- 走到 LLM 兜底，但 LLM prompt（第 89-96 行）也是中文描述
- 实际效果：英文查询几乎全部 fallback 到 synthesis（走完整管线），事实查询无法跳过 HyDE+CRAG 节省 token

### 2.2 代码块已有标注但未独立索引

`backend/app/services/markdown_chunker.py` 已实现代码块保护，产出 `DocumentChunk` 时标注了 `is_code_block=True` 和 `code_language="python"` 等字段。

但 `vector_store.py` 的 `add_chunks` 把代码块和文本块混在一起入同一个 collection，检索时不区分。用户问 "how to write a composable" 时，可能召回的是描述 composables 的文本段落，而非实际的代码示例。

### 2.3 无多跳检索能力

`rag_orchestrator.py` 的 `chat_stream` 是单轮单次检索：
- 查询分类 → HyDE → 混合检索 → RRF → Reranker → CRAG → 生成
- 无法处理 "Vue2 与 Vue3 的响应式系统有什么区别" 这类需要分别检索两个库再综合的查询

## 三、任务分解

### 任务 C1：语义化查询分类

**目标**：查询分类器支持中英文双语，从关键词正则升级为 few-shot embedding 分类，降低 LLM 兜底率与延迟。

**改造文件**：`backend/app/services/query_classifier.py`

**当前实现**（第 38-105 行）：`QueryClassifier.classify` 方法，三阶段：短查询规则 → 中等长度强规则 → LLM 兜底。规则只认中文关键词。

**目标实现**：

1. 正则模式增加英文关键词：
```python
_FACT_PATTERNS = re.compile(
    r"(是多少|哪天|几月|几点|谁|什么人|什么时间|什么时候|在哪|哪里"
    r"|代码|参数|版本|配置|几行|多少页|第几章|具体数字|截至"
    r"|what\s+is|how\s+many|which\s+version|parameter|config|version"
    r"|api\s+reference|return\s+type|default\s+value)",
    re.IGNORECASE,
)
_CONCEPT_PATTERNS = re.compile(
    r"(是什么|什么意思|如何理解|定义|概念|原理|为什么|原因|解释|介绍"
    r"|what\s+does|what\s+is|explain|define|concept|principle|why|reason"
    r"|introduction|overview|understand)",
    re.IGNORECASE,
)
_SYNTHESIS_PATTERNS = re.compile(
    r"(如何|怎么|怎样|比较|区别|vs|对比|分析|总结|归纳|梳理|评估"
    r"|优缺点|利弊|方案|建议|推荐|步骤|流程|方法"
    r"|how\s+to|how\s+do|how\s+can|compare|difference|vs\.?|versus"
    r"|best\s+practice|step\s+by|tutorial|guide|example)",
    re.IGNORECASE,
)
```

2. LLM 兜底 prompt 改为双语：
```python
prompt = (
    "Classify the following developer query into one of three categories:\n"
    "- fact_lookup: queries for specific facts, numbers, parameters, API signatures, return types, version numbers\n"
    "- concept_explain: queries for concept definitions, principles, explanations, overviews\n"
    "- synthesis: queries requiring comparison, analysis, step-by-step guides, best practices, multi-source synthesis\n"
    "Output only the category label, no explanation.\n"
    f"\nQuery: {query}"
)
```

3. 新增 few-shot 示例（嵌入 prompt 或单独维护），覆盖开发者场景典型查询：
```python
 FewShotExample = [
    # fact_lookup
    ("ref() 的返回类型是什么", "fact_lookup"),
    ("what is the default value of props", "fact_lookup"),
    ("FastAPI 的 version 参数", "fact_lookup"),
    # concept_explain
    ("什么是响应式", "concept_explain"),
    ("explain dependency injection in FastAPI", "concept_explain"),
    ("computed 和 watch 的原理", "concept_explain"),
    # synthesis
    ("Vue2 和 Vue3 响应式有什么区别", "synthesis"),
    ("how to migrate from options API to composition API", "synthesis"),
    ("compare ref and reactive", "synthesis"),
]
```
将 few-shot 示例拼入 LLM prompt，提升分类准确率。

4. 新增分类结果缓存（可选，用 LRU dict 缓存 query→label，避免相同查询重复分类）。

**新增配置**（`config.py`）：
```python
QUERY_CLASSIFIER_FEW_SHOT_ENABLED: bool = True
QUERY_CLASSIFIER_CACHE_ENABLED: bool = True
QUERY_CLASSIFIER_CACHE_SIZE: int = 200
```

**验收标准**：
- 查询 "how to use ref()" 命中 synthesis 规则（how to）
- 查询 "what is computed" 命中 concept_explain 规则（what is）
- 查询 "default value of props" 命中 fact_lookup 规则
- 查询 "Vue2 和 Vue3 的响应式有什么区别" 命中 synthesis 规则（区别）
- LLM 兜底率从 ~20% 降到 ~10%（英文查询不再大量 fallback）

### 任务 C2：代码子索引

**目标**：代码块用代码专用 embedding 单独索引，查询含代码意图时路由到代码子索引，提升"这段怎么写"类问题命中率。

**改造文件**：`vector_store.py`、`retrieval_service.py`、`rag_orchestrator.py`、`query_classifier.py`

**当前实现**：`vector_store.py` 单一 collection，代码块和文本块混存。`retrieval_service.search` 不区分代码/文本。

**目标实现**：

#### C2.1 vector_store 新增代码子 collection

`vector_store.py` 新增第二个 collection 用于代码块：

```python
CODE_COLLECTION_NAME = "docs_chat_code"

class VectorStore:
    def __init__(self):
        # ... 原有 self.collection ...
        self.code_collection = self._client.get_or_create_collection(
            name=CODE_COLLECTION_NAME,
            metadata={"hnsw:space": "cosine"},
        )

    def add_chunks(self, chunks: list[DocumentChunk], ...):
        """v4.1: 代码块路由到 code_collection，文本块走原 collection。"""
        code_chunks = [c for c in chunks if getattr(c, "is_code_block", False)]
        text_chunks = [c for c in chunks if not getattr(c, "is_code_block", False)]
        if text_chunks:
            # 原 collection upsert 逻辑
            ...
        if code_chunks:
            # code_collection upsert，用代码专用 embedding（可选）
            self.code_collection.upsert(...)

    def search_code(self, query: str, top_k: int = 5, where: dict | None = None) -> list[dict]:
        """在代码子索引中检索。"""
        results = self.code_collection.query(
            query_texts=[query], n_results=top_k, where=where, ...
        )
        ...

    def get_libraries(self) -> list[dict]:
        """合并两个 collection 的库信息。"""
        ...
```

关于代码专用 embedding：理想情况用 voyage-code-3 或 bge-code，但需额外部署。v4.1 第一步可以先用现有 BGE-M3 embedding（代码块也用同一模型），先验证路由分离的价值，后续再换代码专用模型。在 config 中预留开关：

```python
CODE_EMBEDDING_MODEL: str = ""  # 空=用默认 BGE-M3，非空=代码专用模型
```

#### C2.2 query_classifier 检测代码意图

`query_classifier.py` 新增方法：
```python
def detect_code_intent(self, query: str) -> bool:
    """检测查询是否含代码意图（如"怎么写"/"示例代码"/"how to write"/"code example"）。"""
    code_patterns = re.compile(
        r"(怎么写|示例|代码|写法|用法|写一个|实现)"
        r"|(how\s+to\s+write|code\s+example|sample|implementation|snippet|how\s+do\s+i)",
        re.IGNORECASE,
    )
    return bool(code_patterns.search(query))
```

`classify` 方法返回值从 `str` 改为 `dict`，包含 `query_type` 和 `is_code_query`：
```python
async def classify(self, query, history=None) -> dict:
    query_type = ...  # 原逻辑
    is_code_query = self.detect_code_intent(query)
    return {"query_type": query_type, "is_code_query": is_code_query}
```

#### C2.3 retrieval_service 支持代码检索

`retrieval_service.py` 新增：
```python
def search_code(self, query: str, top_k: int = 5, library: str | None = None) -> list[dict]:
    """在代码子索引中检索。"""
    where = {"library": library} if library else None
    return vector_store.search_code(query, top_k=top_k, where=where)
```

#### C2.4 rag_orchestrator 路由

`rag_orchestrator.py` 的 `chat_stream` 中，分类结果含 `is_code_query=True` 时：
- 额外调用 `retrieval_service.search_code(query, library=library)` 获取代码块
- 将代码块与文本检索结果一起送入 RRF 融合（代码块可适当加权）
- 生成 prompt 中标注代码块来源，引导 LLM 优先使用代码示例

```python
# 在 _parallel_retrieve 之后，RRF 之前
if classify_result.get("is_code_query"):
    code_docs = await asyncio.to_thread(
        retrieval_service.search_code, query, 5, library
    )
    retrieval_lists.append(code_docs)
```

**验收标准**：
- 入库含代码块的文档后，代码块进入 code_collection
- 查询 "how to write a composable" 时，检索结果包含实际代码块
- 查询 "what is reactivity" 时，不触发代码子索引（纯概念查询）
- `vector_store.get_libraries()` 合并两个 collection 的结果

### 任务 C4：Agentic 多跳检索

**目标**：支持跨版本/跨库对比查询（如 "Vue2 与 Vue3 的响应式区别"），通过 query planner 拆解为多步检索再综合。

**前置依赖**：C1 完成（synthesis 分类准确率提升后，才能可靠触发多跳）。

**新增文件**：`backend/app/services/query_planner.py`

**改造文件**：`rag_orchestrator.py`、`config.py`

**当前实现**：`rag_orchestrator.chat_stream` 单轮单次检索，无查询拆解能力。

**目标实现**：

#### C4.1 query_planner 查询规划器

`backend/app/services/query_planner.py`（新建）：

```python
"""查询规划器 —— 将复杂查询拆解为多个子查询。

检测场景：
- 跨库对比："Vue2 和 Vue3 的响应式有什么区别" → 拆为两个子查询
- 跨版本对比："FastAPI 0.100 和 0.99 的差异"
- 多步骤："如何部署 FastAPI 到 Docker 并配置 HTTPS" → 拆为部署 + HTTPS
"""
from __future__ import annotations
import logging
import re
from app.services.llm_service import llm_service
from app.core.config import settings

logger = logging.getLogger(__name__)

class QueryPlanner:
    """检测是否需要多跳，若需要则拆解为子查询。"""

    async def plan(self, query: str, library: str | None = None) -> dict:
        """返回 {"needs_multi_hop": bool, "sub_queries": [{"query": str, "library": str | None}]}。"""
        # 规则优先：检测对比模式
        if self._detect_comparison(query):
            return await self._split_comparison(query, library)
        return {"needs_multi_hop": False, "sub_queries": [{"query": query, "library": library}]}

    def _detect_comparison(self, query: str) -> bool:
        """检测对比类查询。"""
        patterns = re.compile(
            r"(和.{1,15}的?(区别|差异|对比|比较))"
            r"|(vs\.?\s|versus\s)"
            r"|(compare\s+\w+\s+and\s+\w+)"
            r"|(difference\s+between)",
            re.IGNORECASE,
        )
        return bool(patterns.search(query))

    async def _split_comparison(self, query: str, library: str | None) -> dict:
        """LLM 拆解对比查询为子查询。"""
        prompt = (
            "Split this comparison query into two independent sub-queries.\n"
            "Each sub-query should target one side of the comparison.\n"
            "Output JSON: {\"sub_queries\": [{\"query\": \"...\", \"library\": \"...\" or null}]}\n"
            "If a specific library/version is mentioned, set it in the library field.\n"
            f"\nQuery: {query}"
        )
        result = await llm_service.chat(
            messages=[{"role": "user", "content": prompt}],
            system_prompt="You are a query planner. Output valid JSON only.",
        )
        # 解析 JSON，容错处理
        ...

query_planner = QueryPlanner()
```

#### C4.2 rag_orchestrator 集成多跳

`rag_orchestrator.py` 的 `chat_stream` 中，当 `query_type == "synthesis"` 时：

```python
# 在查询分类之后、检索之前
if settings.MULTI_HOP_ENABLED and query_type == "synthesis":
    plan = await query_planner.plan(query, library)
else:
    plan = {"needs_multi_hop": False, "sub_queries": [{"query": query, "library": library}]}

# 对每个子查询分别检索
all_retrieval_lists = []
for sub in plan["sub_queries"]:
    sub_lists = await self._parallel_retrieve(
        [sub["query"]], expand_neighbors=True, library=sub.get("library")
    )
    all_retrieval_lists.extend(sub_lists)

# RRF 融合时合并所有子查询的结果
```

生成 prompt 中增加对比提示：
```python
if plan.get("needs_multi_hop"):
    system_prompt += "\n注意：用户在对比多个主题，请分别检索到的来源进行对比分析，不要混淆。"
```

**新增配置**（`config.py`）：
```python
MULTI_HOP_ENABLED: bool = True
MULTI_HOP_MAX_SUB_QUERIES: int = 3
```

**验收标准**：
- 查询 "Vue2 和 Vue3 的响应式有什么区别" 触发多跳，拆为两个子查询
- 子查询分别检索 Vue2 和 Vue3 的相关内容
- 生成答案中包含对比分析，而非只引用单方面
- 普通查询（"how to use ref()"）不触发多跳，走原单轮路径

## 七、实施顺序

```
C1 语义化查询分类（5 天）
  ↓ （C1 完成后 synthesis 分类准确率提升，C4 才能可靠触发）
C2 代码子索引（7 天，可与 C1 后半并行）
  ↓
C4 Agentic 多跳检索（7 天，依赖 C1 分类准确）
```

C1 是 C4 的前置——多跳触发条件是 `query_type == "synthesis"`，如果分类不准（英文查询全部 fallback 到 synthesis），多跳会误触发。所以 C1 必须先做。
