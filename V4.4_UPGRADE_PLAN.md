## E1: LLM 响应缓存层

### 目标
相同/相似查询直接命中缓存，跳过检索+生成全流程，API 成本降低 60%+，P50 延迟降至 <100ms。

### 现状分析
项目已有 `semantic_cache`（基于 ChromaDB 向量相似度的语义缓存），但存在两个问题：
1. 语义缓存依赖 ChromaDB 向量检索，本身有 ~50ms 延迟
2. 缓存命中率指标已在 `metrics_service` 中统计，但未接入精确缓存（query+library 的 SHA256 精确匹配）

### 实现方案
新建 `backend/app/services/cache_service.py`，实现**两级缓存**：
- **L1 精确缓存**：query+library 的 SHA256 → 结果（TTL 1h，内存字典 + 过期清理）
- **L2 语义缓存**：已有的 `semantic_cache`（向量相似度匹配）

在 `rag_orchestrator.py` 的 `chat_stream()` 入口处：
1. 先查 L1 精确缓存 → 命中则直接返回（<1ms）
2. L1 未命中 → 查 L2 语义缓存 → 命中则返回并回填 L1
3. L2 未命中 → 走完整 RAG 流程 → 结果写入 L1 + L2

### 文件变更
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `backend/app/services/cache_service.py` | 新建 | L1 精确缓存服务，TTL 过期清理 |
| `backend/app/services/rag_orchestrator.py` | 修改 | `chat_stream()` 入口加入 L1 缓存检查 |
| `backend/app/core/config.py` | 修改 | 新增 `CACHE_L1_ENABLED`, `CACHE_L1_TTL_SECONDS` |
| `backend/tests/test_cache_service.py` | 新建 | L1 缓存命中/过期/失效测试 |

### 关键代码结构
```python
# cache_service.py
class CacheService:
    def __init__(self):
        self._store: dict[str, tuple[float, dict]] = {}  # key → (expire_ts, value)
        self._ttl = settings.CACHE_L1_TTL_SECONDS

    def _make_key(self, query: str, library: str | None) -> str:
        return hashlib.sha256(f"{query}|{library}".encode()).hexdigest()

    def get(self, query: str, library: str | None) -> dict | None:
        key = self._make_key(query, library)
        if key in self._store:
            expire_ts, value = self._store[key]
            if time.time() < expire_ts:
                return value
            del self._store[key]
        return None

    def set(self, query: str, library: str | None, value: dict) -> None:
        key = self._make_key(query, library)
        self._store[key] = (time.time() + self._ttl, value)

    def invalidate(self, library: str | None = None) -> None:
        """文档库更新时失效相关缓存"""
```

## E2: SSE 流式体验优化

### 目标
前端实时展示生成过程（检索中→生成中→校验中→完成），打字机效果增强交互体验。

### 现状分析
- 后端 `chat_stream()` 已有事件机制（`yield {"type": "token", ...}`）
- 前端 `useSSE.ts` 已实现 SSE 消费
- 但前端只处理 `token` 事件，未展示检索/校验等阶段状态

### 实现方案
1. **后端**：完善 SSE 事件类型，新增 `stage` 事件（retrieving/generating/faithfulness_check/complete/warning）
2. **前端**：
   - ChatView 消息区域显示阶段指示器（"正在检索文档..." → "正在生成回答..." → "已完成"）
   - token 事件实现打字机效果（逐字显示，可配置速度）
   - warning 事件显示忠实度警告提示条
   - source 事件展示引用来源卡片（可展开查看原文片段）

### 文件变更
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `backend/app/services/rag_orchestrator.py` | 修改 | 在各阶段 yield `stage` 事件 |
| `frontend/src/composables/useSSE.ts` | 修改 | 解析 `stage` 事件类型 |
| `frontend/src/views/ChatView.vue` | 修改 | 阶段指示器 + 打字机效果 |
| `frontend/src/components/ChatMessage.vue` | 修改 | 引用来源卡片展开 |

## E3: Docker 容器化部署

### 目标
一键 `docker-compose up` 启动完整服务，降低复现成本，展示部署运维能力。

### 实现方案
- **后端 Dockerfile**：Python 3.11-slim + 依赖安装 + uvicorn 启动，暴露 8000 端口
- **前端 Dockerfile**：Node 20-alpine + npm build + nginx 静态托管，暴露 80 端口
- **docker-compose.yml**：编排前后端服务，网络互通，环境变量通过 `.env.docker` 注入
- **健康检查**：后端 `/health` 端点，前端 nginx 健康检查

| `frontend/nginx.conf` | 新建 | nginx 配置（SPA 路由 + API 反代） |
| `docker-compose.yml` | 新建 | 服务编排 |

